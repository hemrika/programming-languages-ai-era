# Formal v0.3 Review Pass

*Date: 2026-04-30. Reviewer: Rutger Hemrika.*

This document executes the Phase 7 review checklist against the v0.3 framework state: 10 languages, 7 dimensions, half-point scoring, greenfield framing. Each checklist item is walked with status (PASS / NEEDS-WORK / FAIL), evidence pointer, and recommended action. The review surfaces score-adjustment candidates and counterclaim gaps but does not itself change scores — any adjustment lands in a separate commit.

## Scope

The review covers framework documents (`framework/*.md`), claims (`claims/<lang>.yaml`), evaluations (`evaluations/<lang>.yaml`), comparisons (`comparisons/*.md`), insights (`insights/*.md`), and the report outputs. The validator run prior to review was clean: `All evaluations valid. 114 counters links validated.`

## Checklist (greenfield-adjusted)

### 1. Are sources current enough?

**Status: PASS with one watchpoint.**

Spot-checks across the 14 source registries (`sources/<lang>-sources.yaml`):

- TypeScript, Python, .NET, Rust, Go, Kotlin, Swift, Elixir, Java, C++ — each has a mix of 2024–2026 publication dates for the AI-systems and structured-output sources, with older-but-stable references for spec/design documents (e.g. PEP 484 from 2014, ISO C++ 2020 standard).
- The institutional safety-pressure sources (NSA *Software Memory Safety* 2022, ONCD *Back to the Building Blocks* 2024, MSRC ~70% memory-safety figure 2019, Chromium ~70% figure 2020) are the oldest in the corpus but the recommendations have been reaffirmed in subsequent guidance. No update needed.
- One watchpoint: the structured-output dimension was added 2026-04-30 and the underlying libraries (Pydantic v2, Zod, Instructor, Outlines, Vercel AI SDK) move quickly. A 6-month refresh of these sources is reasonable.

**Action:** none required. Schedule a structured-output source refresh for Q4 2026.

### 2. Are official claims overtrusted?

**Status: PASS.**

The schema includes `evidence_type` and `confidence` per claim, plus a `counters` field linking counterclaims to positive claims. Spot-checks:

- Every Python `evidence_type: official_documentation` claim with `confidence: high` (e.g., python-001 dynamic typing, python-008 ML stack) has at least one counterclaim with `polarity: negative` (python-019 type hints not enforced; python-020 stub fragmentation).
- Every TypeScript official claim has structural counterclaims (typescript-005 unsoundness; typescript-017 escape hatches).
- Rust official claims are paired with counterclaims for unsafe boundaries (rust-011, rust-012).

The `counters:` validator confirmed 114 counter-links resolve cleanly. The pattern is consistent across the 10 languages.

**Action:** none required.

No conflation observed. The dimension field forces separation at the schema level.

**Action:** none required.

### 5. Are dynamic-language advantages fairly represented?

**Status: PASS.**

The v0.3 ranking has Python at #2 (4.25) and TypeScript at #1 (4.38). Both score ≤4 on machine_cognition (3 and 4 respectively) but lead on:

- AI-systems interoperability: Python 5.0, TypeScript 4.5 — credited explicitly via Pydantic, Anthropic+OpenAI SDKs, LangChain Python+JS, MCP SDKs, vector-store SDKs, ONNX bindings.
- Structured-output maturity: Python 5.0, TypeScript 5.0 — credited via Pydantic, Instructor, Outlines, Zod, Vercel AI SDK.
- Strategic viability: both 4 — credited for sustained typing-PEP cadence (Python) and Microsoft stewardship + corpus density (TypeScript).

The gradual-typing position is captured at machine_cognition rather than dismissed. `comparisons/dynamic-vs-static.md` makes the case explicit.

**Action:** none required.

### 6. Are legacy advantages improperly credited?

**Status: PASS.**

(This item is the greenfield-inverted form of the original "are legacy-language advantages fairly represented?". Under greenfield framing we ask whether legacy is improperly credited.)

Spot-checks of the languages where the temptation is highest:

- **Java:** runtime_ecosystem 4.5 — justified by virtual threads, JEP 453 structured concurrency, Maven Central as a forward distribution channel. Strategic viability dropped from 4 to 3 explicitly because "the durable adoption premium that previously lifted Java to 4 is not credited." Greenfield discipline holds.
- **.NET:** runtime_ecosystem 4 — justified by cross-platform runtime + performance, not historical Windows installed base. Strategic viability 4 is justified by Microsoft AI-tooling alignment, not by enterprise inertia.
- **C++:** runtime_ecosystem 4 — explicitly defended in `comparisons/lens-analysis.md` as forward credit for the production toolchain (Clang, GCC, MSVC), ABI stability, and accelerator-host-language status (CUDA, ROCm/HIP, SYCL). Strategic viability 2.0 — driven by NSA/ONCD/MSRC/Chromium/Android safety pressure, with the safety penalty knowingly accepted.

The framework explicitly distinguishes ecosystem-velocity (forward) from ecosystem-viability (forward production-relevance) from installed-base (excluded). Greenfield discipline holds.

**Action:** none required.

### 7. Are safety claims scoped correctly?

**Status: PASS.**

Spot-checked Rust and C++ safety claims:

- Rust: rust-001 ownership system at compile time, rust-002 single-owner semantics, rust-003 lifetime-checked references — all scope-restricted to safe Rust by the notes field. The unsafe boundary is explicitly captured by rust-011 (`unsafe` block scopes), rust-012 (FFI exit). These are linked as counterclaims to the positive safety claims.
- C++: cpp-014 (accelerator dialects diverge from ISO C++), cpp-015 (modules adoption uneven), cpp-016 (Core Guidelines advisory) — all scoped as counterclaims to cpp-001 (ISO standardization). The safety-pressure thesis cites memory-safety percentage figures with their sources (MSRC, Chromium, NSA, ONCD, Android telemetry), each with a publication date and direct source link.

No overclaiming observed. Safety claims are correctly bounded.

**Action:** none required.

### 8. Are scores comparable across languages?

**Status: NEEDS-WORK on three cells; PASS overall.**

Cross-language spot-check on same-score cells:

- HC=4 across TypeScript, Python, Kotlin, .NET, Swift, Elixir — these are consistent but the justifications differ in flavor. The rubric in `framework/scoring-rubric.md` provides the anchor, but the rater could move any of these by 0.5 in either direction without difficulty. Confidence is appropriately set at "medium" for all six.
- AO=3 across Java, Swift, Elixir — these are also defensible at 3.5. Java's training-corpus density argues for an upward adjustment; Swift's Apple-platform tooling depth argues for an upward adjustment; Elixir's BEAM-model unfamiliarity argues for stability. **Recommendation: Java AO 3 → 3.5 in a future review.**
- AI=3 across Rust, Kotlin — Rust is closer to 3.5 given the production-grade Postgres/Kafka clients and candle inference runtime. Kotlin is closer to 3 given JVM-interop dependence. **Recommendation: Rust AI-systems 3 → 3.5 in a future review.**

These are not fatal — the relative order is preserved — but the rubric anchoring could be tightened.

**Action:** Two score-adjustment candidates surfaced (see "Score adjustments recommended" below). No commit yet.

## Counterclaim gaps to close

Spot-check surfaced two cells where positive claims lack a documented counterclaim:

1. **Python ai_systems_interoperability (5.0):** Currently no counterclaim flagging the GIL constraint on inference-serving workloads. Worth adding a counterclaim along the lines of "Python's GIL limits inference-serving concurrency relative to Go or Rust; production deployments commonly proxy through Go/Rust/Triton servers."

2. **Kotlin structured_output_maturity (3.5):** No counterclaim flagging that the LLM-specific structured-output ergonomics layer is thinner than Python or TypeScript. The justification mentions this in passing but no formal counterclaim is filed.

These are documentation gaps, not score errors. Adding counterclaims would not change the scores.

## Overall verdict

The v0.3 framework holds up to the Phase 7 checklist. No item failed. Two cells (Java AO, Rust AI-sys) surfaced score-adjustment candidates that should be considered in the next review cycle. Two counterclaim gaps were identified for documentation completeness. The greenfield discipline is consistently applied across the 10 languages — incumbents (Java, .NET, C++) are not over-credited and forward-only properties (Python typing PEPs, TypeScript `satisfies`, Elixir set-theoretic types) are credited at verification velocity.

The framework is **review-passed** at v0.3.
