# Incumbent Risk

## Thesis

Under AI-era pressure - memory-safety regulation, verification velocity, and AI-agent operability - incumbent languages whose forward case rests primarily on installed base face structural risk that a greenfield framing exposes. Legacy gravity slows displacement but does not insulate against three forces acting on language choice for *new* projects: regulatory selection against unsafe defaults, agent-operability gaps in fragmented toolchains, and verification-ceiling competition from modern typed languages. C++ is the clearest case; Java is milder; Python sits at a boundary. This is falsifiable: if installed base alone were sufficient to carry forward strategic viability, removing legacy credit from the framework would not have moved any incumbent's score.

## Evidence

C++ presents the cleanest case. Forward-looking structural credit remains: ISO standardization with three-yearly revisions [cpp-001], C++20 modules and C++17 features as language-level additions [cpp-002, cpp-010], the Core Guidelines [cpp-003], and - the strongest forward signal - the dominant accelerator toolchains (CUDA, ROCm/HIP, SYCL, oneAPI) all primarily targeting C++ as their host language [cpp-013]. These keep runtime/ecosystem at 4. But four documented sources of regulatory and platform-owner pressure sit on the negative side: MSRC reporting ~70 percent of CVEs are memory-safety issues across products written largely in C and C++ [cpp-004], Chromium reporting the same ~70 percent figure for high-severity bugs [cpp-005], the NSA recommending a shift away from memory-unsafe languages [cpp-006], and the ONCD's 2024 *Back to the Building Blocks* report [cpp-007], with Google's Android telemetry showing memory-safety vulnerabilities declining as new native code shifted away from C and C++ [cpp-008] toward memory-safe alternatives including Rust [rust-016, rust-017]. Counter-claims [cpp-014, cpp-015, cpp-016] track that the accelerator dialects diverge from ISO C++, modules adoption remains incomplete, and the Core Guidelines are advisory. Strategic viability scores 2 and the weighted total reaches 2.65.

Java is the milder case. Forward credit holds for virtual threads as a JVM-level concurrency primitive [java-006], Maven Central as a versioned artifact channel [java-010] - explicitly reframed under greenfield as forward ecosystem viability rather than installed-base credit - structured concurrency in JEP 453 [java-015], the JVM's portable bytecode and verifier checks [java-009], and JDT.LS for AI tooling integration [java-014]. The OpenJDK release model with six-month cadence [java-007] and backward compatibility commitments [java-008] are reframed as governance rather than adoption-base premium. The negatives [java-016, java-017, java-018] track real costs: preview features stabilize across multiple releases delaying LTS availability, single-inheritance raises ceremony, and modern and legacy idioms coexist in the standard library. JVM-native modern languages with stronger structural typing - Kotlin's null-safety and sealed types [kotlin-001, kotlin-002] - exemplify the verification competition Java faces from inside its own ecosystem.

Python sits at the boundary. Forward credit is strong: top GitHub language by activity as a training-corpus signal [python-012], top tier of Stack Overflow professional usage [python-013], the typing PEP cadence [python-016] and PEP 484 syntax [python-003], pyright/Pylance with bundled stubs [python-018], the annual release cadence per PEP 602 [python-017], and uv as a Rust-based fast package manager [python-011]. But verification weakness is load-bearing: type hints are not enforced at runtime [python-002, python-019], typing benefits fragment across the dependency graph where stub coverage is incomplete [python-020], and dynamic typing means many type errors surface only at execution time [python-015]. Python keeps runtime/ecosystem at 5 because ML-domain library velocity is forward-relevant, but drops out of the top tier when installed-base credit is removed. The contrast with TypeScript - a real static type layer over a dynamic substrate [typescript-001] with discriminated unions [typescript-002] - illustrates the verification-velocity competition Python faces.

## Counter-positions

The strongest counter is that production realities still need software to *run*. Ecosystem viability for incumbents - C++'s accelerator toolchains [cpp-013], the JVM's portable bytecode and Maven Central [java-009, java-010] - is forward-relevant for greenfield projects whose deployment and hardware integration depend on these stacks. The framework explicitly credits this through ecosystem viability. Greenfield C++ host code for accelerator kernels and JVM services for high-throughput backends remain defensible, with the safety penalty knowingly accepted in the C++ case. Go's deliberate minimalism [go-009] and unified toolchain [go-003] illustrate that a non-incumbent forward profile is now the standard greenfield offer, not a niche alternative. The thesis is not that incumbents collapse but that their forward case becomes domain-specific, not default.

## Implications

For greenfield AI-era language choice, incumbent gravity should not drive selection. Teams should evaluate against the matrix in `comparisons/overview.md` rather than defaulting to the language they know best. The portfolio note in `comparisons/lens-analysis.md` is concrete: C++ is a default only for accelerator host code, Java a default for none of the listed greenfield categories, Python a default for data and ML-leveraged work where verification gaps are acceptable.
